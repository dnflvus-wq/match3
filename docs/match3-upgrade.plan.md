# Match3 게임 고도화 계획서

## 현재 문제점
1. 스왑 시 타일이 시각적으로 안 따라옴
2. 매치 실패 시 돌아오는 애니메이션 버그 (블록 사라짐)
3. 힌트 시스템 없음 (아무 타일이나 흔들림)
4. 게임 필(Juice)이 전혀 없음 — 터지는 이펙트, 소리, 화면 흔들림 등
5. 게임 상태 관리 없음 — 입력이 아무 때나 먹혀서 꼬임

## 구현 순서 (우선순위)

### Phase 1: FSM (유한 상태 머신) 도입 — 가장 먼저
**왜:** 다른 모든 기능의 기반. 상태 관리 없이는 애니메이션/힌트 전부 꼬임

**상태 정의:**
```
PREGAME → READY → SWAPPING → CHECKING → MATCH → COLLAPSE → READY (루프)
                                          ↓ (매치 없음)
                                        READY
```

- `READY`: 유저 입력 대기 (이 상태에서만 터치 가능)
- `SWAPPING`: 스왑 애니메이션 재생 중 (입력 차단)
- `CHECKING`: 매치 여부 판별
- `MATCH`: 매치된 타일 제거 애니메이션
- `COLLAPSE`: 빈 칸 채우기 + 새 타일 스폰
- `ENDGAME`: 게임 종료

**구현 방법:**
- GameGrid.cs에 `enum GameState` 추가
- Update()에서 현재 상태에 따라 분기
- 입력은 READY 상태에서만 처리
- 각 상태 전환 시 코루틴으로 애니메이션 완료 대기

### Phase 2: 스왑 애니메이션 개선
**현재:** Move() → 즉시 X,Y 변경 + 선형 보간
**목표:** Ease-Out 보간 + 핑퐁(실패 시 갔다 돌아오기) + Z축 깊이 조절

**구현:**
1. MoveCoroutine에 Ease-Out 함수 적용: `t = 1 - (1-t)*(1-t)` (Quadratic Ease-Out)
2. 실패 시 핑퐁: SWAPPING → CHECKING → 매치 없음 → 역방향 스왑 애니메이션 → READY
3. Z축: 스왑 중 한쪽 타일을 z=-0.1로 내려서 겹침 방지
4. **논리적 위치(배열)와 시각적 위치(Transform) 완전 분리**

### Phase 3: 힌트 시스템
**알고리즘:** 브루트 포스 (8x8 = 최대 112가지, 성능 문제 없음)
1. 보드 전체 순회
2. 각 타일을 상하좌우로 가상 스왑
3. 스왑 후 매치 3 이상 되는지 확인
4. 발견하면 해당 타일 좌표 반환

**시각적 힌트:**
- 5초 대기 후 매치 가능 타일에 크기 펄스 애니메이션 (Sin 함수)
- 유저가 터치하면 즉시 초기화

### Phase 4: Juicing (게임 필)
1. **Squash & Stretch**: 타일 떨어질 때 찌그러짐 → 원복
2. **파티클**: 매치 시 반짝이/별 이펙트 (오브젝트 풀링)
3. **Screen Shake**: 4매치 이상 또는 콤보 시 카메라 미세 흔들림
4. **오디오 피치**: 콤보 연속 시 효과음 피치 점진 상승
5. **히트 스톱**: 대형 매치 시 0.05초 일시 정지
6. **절차적 애니메이션**: Idle 상태 타일 미세 호흡 (Sin 함수)

### Phase 5: 추가 기능
- 특수 타일 이펙트 강화 (줄무늬, 폭탄, 레인보우)
- 콤보 카운터 UI
- 레벨 클리어 연출 개선

## 추가 구현 세부사항 (유니티 개발 노트북 리서치)

### 오브젝트 풀링 상세
- 색상별로 별도 풀 관리 (빨강 풀, 파랑 풀 ...)
- CPU 스파이크가 안 보이는 시점(메뉴 화면)에서 사전 할당
- 터질 때 Destroy 대신 비활성화 → 풀에 반환 → 재사용 시 활성화

### ScriptableObject 레벨 데이터
- 각 레벨을 ScriptableObject로 생성
- 포함 데이터: 그리드 레이아웃, 목표, 이동 횟수, 색상별 스폰 가중치
- 커스텀 에디터로 시각적 레벨 디자인 가능

### T자/L자 매치 감지 (재귀 플러드필)
1. 보드 배열 복사본(toTest) 생성
2. 미방문 타일을 시드로 선택
3. 상하좌우 4방향 재귀 탐색 → 같은 색상이면 Collector 리스트에 추가
4. 방문한 타일은 null 처리 (중복 방지)
5. Collector에 3개 이상이면 매치 성공

### 이징 함수
- DOTween 도입 권장 (Elastic, Back, Bounce 이징)
- DOTween 없이 하려면 Mathf.Sin 기반 커스텀 이징 함수 직접 구현
- 스왑: 0.25초, 파괴: 0.15초 권장

### FSM 상세 흐름 (두 번째 리서치 보강)
```
READY → MOVE(스왑 애니메이션) → EVALUATE(매치 스캔)
  ↑        ↓ 매치 없음: 거절 애니메이션 → READY
  ↑        ↓ 매치 있음:
  ↑     MATCH(타일 제거 + 특수 폭탄 생성)
  ↑        ↓
  ↑     COLLAPSE(새 타일 스폰 + 낙하)
  ↑        ↓ 재매치 있음 → MATCH (연쇄)
  ↑        ↓ 재매치 없음 + 유효 이동 있음 → READY
  ↑        ↓ 유효 이동 없음 → SHUFFLE → READY
  └────────┘
```
- READY가 아닐 때 'Processing' 플래그로 입력 차단
- 애니메이션 완전 종료 후에만 다음 상태 전이

## 코드 리팩토링 (Phase 1 전제)

**현재 구조 (God Class 문제):**
- GameGrid.cs에 모든 로직 집중 (입력, 매치, 스폰, 애니메이션 전부)
- MovablePiece.cs의 Move()가 즉시 X,Y 변경 → 데이터-시각 꼬임 원인

**목표 구조 (MVC 분리):**
```
GameController.cs (기존 GameGrid.cs 리팩토링)
├── FSM 상태 관리
├── 입력 처리 (READY 상태에서만)
├── Model ↔ View 중재
│
BoardModel.cs (순수 C# 클래스, MonoBehaviour 아님)
├── 2차원 타일 배열 데이터
├── 매치 검사 알고리즘 (가로/세로 스캔)
├── 타일 드롭(중력) 계산
├── 힌트 탐색 (가상 스왑 시뮬레이션)
│
BoardView.cs + PieceView.cs (시각 전담)
├── 타일 생성/파괴 (오브젝트 풀링)
├── 스왑 애니메이션 (Ease-Out 코루틴)
├── 파티클, Screen Shake, 오디오
```

## 핵심 함정 & 방지법 (리서치 결과)

### 1. Unity 물리 엔진 사용 금지
- 타일 낙하에 Rigidbody/Collider 절대 쓰지 말 것
- 부동 소수점 오차로 기기마다 결과 다름 → 직접 좌표 계산 + 커스텀 Tween

### 2. Instantiate/Destroy 반복 금지
- GC 스파이크 → 프레임 드랍, 크래시
- 게임 시작 시 오브젝트 풀 생성 → 비활성화/활성화로 재사용

### 3. 초기 보드 매치 방지
- 타일 배치 시 왼쪽 2개 + 아래 2개 검사
- 동일 색상 연속 2개면 해당 색상 배제

### 4. 데드 보드 방지
- 보드 생성 후 유효한 이동이 1개 이상 있는지 검사
- 없으면 자동 셔플

### 5. 새 타일 스폰 가중치
- 순수 랜덤 → 연쇄 재앙 발생 가능
- 현재 보드에 많은 색상은 등장 확률 낮추기 (동적 가중치)

### 6. 모바일 터치
- Screen → Tile 좌표 변환 정확히
- 드래그 임계값 설정 (타일 절반 길이 이상 스와이프)
- 매치 실패 시 거절 애니메이션 필수 (무시 금지)
