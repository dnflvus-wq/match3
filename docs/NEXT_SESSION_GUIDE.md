## 💡 핵심 교훈 (사용자 + Claude 공동 성찰)

### 1. 오픈소스 구조는 원래 정상이었다
- 원본 GameGrid.cs는 God Object였지만 **잘 돌아갔다**
- 5개 레벨 규모에서는 문제없는 구조였다
- MVC로 쪼갠 건 "1000개 레벨, 특수타일, 고도화"를 위한 것
- **돌아가는 걸 4일간 건드려서 망가뜨린 셈**

### 2. Model 순수 C#은 올바른 방향이다
- 사용자가 처음엔 "왜 순수 C#으로 만들었냐"고 화냈지만
- 실제로는 **맞는 패턴**이었다
- 웹으로 치면 Service 레이어가 Spring에 의존 안 하는 것과 같음
- 테스트 가능 + 재사용 가능
- **문제는 Model이 아니라 View/에디터 연동을 안 한 것**

### 3. Unity는 에디터를 써야 한다
- 계획서에도 "에디터에서 할 수 있는 것은 에디터에서"라고 적혀있었음
- MCP까지 깔아줬는데도 코드로만 작업
- **프리팹 안 쓰고 동적 생성 → UI 엉망**
- **Inspector 설정 안 하고 하드코딩 → 레벨 통일 안 됨**
- **씬 편집 안 하고 코드로 레이아웃 → 에디터/플레이 모드 불일치**

### 4. 화면 구분
| 영역 | 방식 | 예시 |
|------|------|------|
| HUD, 메뉴, 팝업 | **Unity UI (Canvas)** | 점수판, 게임오버 화면 |
| 게임 보드, 타일 | **SpriteRenderer (월드)** | 6x6 타일 격자 |

### 5. 검증 없이 "완료"는 거짓말
- 컴파일 성공 ≠ 게임 동작
- State=READY ≠ 실제 플레이 가능
- **UI까지 MCP로 스크린샷 확인해야 완료**
- 4일 삽질의 근본 원인

### 6. 감독자(PM 클로드)의 한계
- cmd 클로드 감독 자동화 시도 → 실패
- 2분마다 체크한다고 했지만 50분 놀기도 함
- 멈춘 cmd 클로드를 재시작 안 하고 기다리기만 함
- **사용자가 직접 개입해야 돌아감**
- 사용자 지식이 부족하면 검증이 안 되고, 검증 안 되면 빈 껍데기만 쌓임

### 7. 사용자와 Claude의 한계
- **사용자 한계**: Unity 전문가가 아님 (BE 개발자 배경)
- **Claude 한계**: 검증/감독 능력 부족, 시키는 것만 함
- **해결**: 사용자가 화면 확인 + Claude가 MCP로 에디터 조작 + 서로 피드백

## 🎯 다음 세션 방향 (사용자 판단)

### 결정: **처음부터 새로 설계**
4일 삽질 끝에 내린 결론 — 기존 프로젝트 리팩토링은 한계.

### 새 프로젝트 전략
```
1. 새 Unity 프로젝트 생성
2. 처음부터 MVC로 설계
   - Model: TileData[,] 기반 순수 C# (GamePiece[,] 아님)
   - View: Prefab + SpriteRenderer (코드 동적 생성 금지)
   - Controller: FSM만 담당 (130줄 이하)
3. 기존 오픈소스에서 복사할 것
   - 매치 탐지 알고리즘 (MatchFinder)
   - 낙하/클리어 로직 (BoardModel)
   - 특수타일 효과 (RowClear, ColumnClear, Rainbow)
4. 새로 만들 것
   - 씬 1개 + LevelConfig ScriptableObject 기반
   - HUD, GameOver, LevelSelect 전부 Prefab으로 배치
   - Inspector에서 값 설정 (하드코딩 금지)
```

### 예상 공수
- 새 프로젝트 세팅 + MVC 뼈대: 2시간
- 기존 로직 이식 + 테스트: 3시간
- 에디터 작업 (프리팹, 씬, UI): 3시간
- Play 테스트 + 버그 수정: 2시간
- **총 10시간 (사람 기준)** / **바이브코딩 4~6시간**
